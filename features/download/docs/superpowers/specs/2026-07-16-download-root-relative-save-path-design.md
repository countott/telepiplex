# download 根目录相对保存路径设计

## 背景与范围

Open115 1.2.0 的 Telegram 保存目录配置要求用户发送以 `/` 开头的 115 绝对路径。Telepiplex 在 Feature 会话等待文本输入时会拦截所有以 `/` 开头的消息，避免把 Telegram 命令误交给会话，因此 `/Series` 一类合法目录输入无法到达 Open115。

本次只修改 `feature/download` 的保存目录路径合同、交互文案、配置 Schema、测试和版本。Telepiplex 的命令路由规则保持不变；当前 `save_directories` 为空，不增加旧绝对路径兼容或迁移逻辑。

## 路径合同

`save_directories[].path` 改为从 115 根文件夹开始的相对路径，不得以 `/` 开头。以下输入有效：

```text
series/live action
series/live action/
```

两种写法统一保存为：

```text
series/live action
```

路径按以下顺序校验和规范化：

1. 去除整段输入首尾空白；
2. 允许并移除一个末尾 `/`；
3. 拒绝空值、以 `/` 开头的值、连续 `/`（包括重复尾斜杠）形成的空路径段；
4. 拒绝值为 `.` 或 `..` 的路径段；
5. 使用规范化后的路径检查重复；
6. 保留目录名称的大小写和路径段内部空格。

`config.schema.json` 使用同一合同约束直接编辑配置文件的用户。Feature 启动时也使用同一规范化函数检查磁盘配置：单个尾斜杠会被移除并写回；规范化后重复的目录会阻止启动。运行中配置和写回磁盘的路径始终使用无起始、无结尾 `/` 的规范形式。

## Telegram 交互

新增和编辑保存路径时，提示文案改为要求“从 115 根文件夹开始填写”，示例使用 `series/live action`，并说明末尾 `/` 可省略。用户如果发送以 `/` 开头的值，即使在绕过 Telegram 路由的直接调用中，也会收到“不需要以 / 开头”的明确错误。

显示目录详情、删除确认和目录列表时使用配置中保存的根目录相对路径，不再向用户展示为必须以 `/` 开头的格式。名称的新增、编辑、删除、分页、暂存和原子保存行为不变。

## 下载数据流

目录配置保持根目录相对路径。用户在 `/magnet` 中选择目录后，现有下载入口 `_start_download` 继续在调用 115 客户端前统一构造以 `/` 开头的 API 绝对路径。115 客户端的 `_normalize` 仍作为最后一道边界规范化，因此对 115 API 的请求合同不变。

这使 Telegram 输入合同与 115 API 合同分离：用户发送 `series/live action`，配置保存 `series/live action`，下载执行时传给 115 客户端 `/series/live action`。

## 错误处理与原子性

路径校验失败时保留当前目录编辑阶段和工作副本，不写磁盘。规范化后的重复路径继续使用现有重复提示。点击“保存并完成”后仍只原子替换 `save_directories`，保留 Token、授权模式和其他配置字段；写入失败继续保留工作副本供重试。

## 测试与发布

测试先复现当前代码拒绝 `series/live action`、接受 `/Series` 的旧行为，再实现新规则。覆盖以下行为：

- 新增和编辑接受无起始 `/` 的多级路径；
- 末尾 `/` 被移除并以规范路径写回；
- 以 `/` 开头、空路径段、`.`、`..` 和规范化后的重复路径被拒绝；
- 配置 Schema 与运行时路径合同一致；
- 选择相对目录后，下载边界仍向 115 客户端传递绝对路径；
- Open115 全量测试、编译和差异检查通过。

初始实现曾发布不可变版本 `1.2.1`。发布后代码审查发现重复尾斜杠和启动时未规范化磁盘配置两个边界问题，因此最终修正版提升为 `1.2.2`。完成验证后提交并推送 `feature/download`，在 Telepiplex 发布基础设施提交上创建并推送 `download-v1.2.2`，等待 GitHub Actions 完成，再验证公开 Release、Linux `.tpx`、SHA-256、来源提交和 `catalog` 分支条目。
