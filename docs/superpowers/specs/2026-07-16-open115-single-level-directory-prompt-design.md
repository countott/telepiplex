# Open115 单级保存目录交互澄清设计

## 背景与根因

现场日志显示，Open115 1.2.1 在“新增目录”流程中先等待显示名称，再等待实际保存路径。用户发送 `真人电影` 后，Feature 已将它成功接收为显示名称并进入 `directory_add_path`；由于随后点击“返回”，`真人电影` 从未进入路径校验。第二次同样先把 `真人电影/` 接收为显示名称，直到收到 `真人电影/movie` 才完成路径步骤。

现有路径规范化、Schema 和已发布 1.2.2 包均允许单级根目录相对路径 `真人电影`。问题不是路径能力缺失，而是两步交互只给出了多级路径示例 `series/live action`，没有明确说明显示名称不是路径，也没有给出单级路径示例，导致用户误认为必须提供两级目录。

## 交互合同

保留 `name` 与 `path` 分离的数据模型和两步流程，不改变已有配置格式：

1. 新增目录第一步明确提示“显示名称只用于按钮展示，不是保存路径”，示例为 `真人电影`。
2. 新增目录第二步明确提示“单级目录直接输入 `真人电影`；多级目录可输入 `series/live action`；不要以 `/` 开头；末尾 `/` 可省略”。
3. 编辑名称和编辑路径使用同一术语，避免新增与编辑出现不同合同。
4. 单级输入 `真人电影` 与 `真人电影/` 均规范化保存为 `真人电影`；多级路径合同保持不变。

用户要配置一个根目录文件夹时，完整输入序列为：

```text
第一步（显示名称）：真人电影
第二步（保存路径）：真人电影
```

最终配置为：

```yaml
- name: 真人电影
  path: 真人电影
```

## 实现边界

只修改 Open115 Feature 的交互文案、回归测试、README 和版本。`directories.py`、`config.schema.json`、Core 消息路由及下载 API 边界保持不变，因为这些层已经正确接受单级相对路径。

## 测试与发布

先修改现有目录编辑端到端测试，使其要求新增提示明确区分两步，并实际以 `真人电影` 作为显示名称和单级路径完成暂存、保存与 `/magnet` 目录展示。测试必须先因旧提示不含单级说明而失败，再进行最小文案修改。

版本提升为 `1.2.3`。全量测试、编译、差异检查和代码审查通过后，推送 `feature/open115`，创建 `open115-v1.2.3` 标签并验证 GitHub Actions、公开 Release、Linux `.tpx`、SHA-256、来源提交和 `catalog` 条目。
