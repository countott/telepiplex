# open115 保存目录可视化配置设计

## 范围

本次只修改 `feature/open115`。`save_directories` 继续使用现有结构：

```yaml
save_directories:
  - name: 剧集
    path: /Series
```

`config.default.yaml` 与 `config.schema.json` 已声明该字段，本次新增 Telegram 可视化管理流程、原子持久化能力、运行时同步和对应测试，不改变下载或存储能力契约。

## 入口与导航

`/config` 改为 open115 配置首页，展示当前授权状态和保存目录数量，并提供“授权配置”“保存目录”“退出”三个入口。“授权配置”复用现有 Access/Refresh Token 与扫码授权流程；`/auth` 仍直接进入现有授权方式选择，不增加目录步骤。

“保存目录”进入分页列表。每个目录使用稳定的会话内索引作为回调参数，点选后进入单项菜单，可编辑名称、编辑路径或删除。列表同时提供“新增目录”“保存并完成”“退出不保存”；目录数量超过单页容量时提供上一页和下一页，确保生成的键盘不超过 Core 支持的十行限制。

## 编辑与校验

目录管理会话创建现有 `save_directories` 的深拷贝作为工作副本。新增时依次收集显示名称和 115 绝对路径；编辑时只收集被选中的字段；删除前要求显式确认。每次操作完成后返回目录列表，但不立即修改磁盘或运行中配置。

名称必须是非空单行文本；路径必须是非空单行文本并以 `/` 开头。名称和路径分别在整个工作副本内保持唯一；编辑当前条目时不与自身冲突。输入无效、重复或步骤与会话状态不匹配时保留当前工作副本，并提示重新输入或返回列表。

点击“保存并完成”后，向 Feature 私有配置原子写入完整 `save_directories` 列表。写入成功后用返回的新配置同步 `Open115Feature.config`，后续 `/magnet` 立即使用新目录，不重启 Feature。点击“退出不保存”、发送 `/q` 或会话超时都丢弃工作副本，不写配置。

## 持久化与失败处理

`FeatureConfigStore` 新增专用的保存目录写入方法。该方法在锁内读取当前完整配置，仅替换 `save_directories`，再复用现有 `0600` 临时文件和原子替换逻辑，因此保留 Token、授权模式及其他字段。

写入失败时不更新运行中配置，也不关闭目录管理会话；用户可重试保存或退出。响应和日志只报告异常类型，不暴露 Token 或其他配置值。删除、编辑和新增在最终确认前均只作用于工作副本，因此退出不会留下部分修改。

## 操作状态

`/config` 目录管理沿用 open115 的 operation 状态模型。进入目录页和等待文本输入时为 `awaiting_input`；确认保存时为 `running/config_persistence`；成功后为 `completed`。退出或超时为 `cancelled`。保存失败恢复为 `awaiting_input`，并保留工作副本供重试。

## 测试与文档

测试覆盖：

- `/config` 首页与 `/auth` 原入口的职责分离；
- 现有目录分页展示和回调命名空间；
- 新增、编辑名称、编辑路径、确认删除；
- 空值、换行、非绝对路径、重复名称和重复路径校验；
- 退出、`/q` 与超时不写配置；
- 保存时只替换 `save_directories`，保留 Token 和其他字段，并保持文件权限；
- 保存成功后运行时立即读取新目录，保存失败则保留旧配置和工作副本；
- 现有 Token 与扫码授权流程无回归。

README 更新 `/config` 说明，明确保存目录可在 Telegram 中逐条新增、编辑和删除，最终统一确认保存。
